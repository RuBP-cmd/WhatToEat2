# WhatToEat2 项目 AI 辅助开发约束规范

## 1. 项目概览与核心业务
本项目 `WhatToEat2` 是一款基于 Compose Multiplatform 开发的跨端应用（同时支持 Android 与 iOS）。
其核心功能包括：获取云端存储的点评以及食物数据（包含评分、图片、标签、位置），按食物名称和tag搜寻食物，对食物进行点评，以及实验网站快捷入口、随机抽取一个食物的一些小功能。

## 2. 目录结构与隔离红线
- **主战场 (`commonMain`)**：所有的业务逻辑、网络请求、数据持久化以及 Compose UI 必须写在 `commonMain` 源码集中。
- **绝对红线**：严禁在 `commonMain` 内导入任何 `java.*` 或 `android.*` 前缀的包。如果必须调用底层硬件能力，强制要求使用 Kotlin 的 `expect/actual` 机制进行抽象。
- **宿主结构**：应用的 Android 端入口以 `MainActivity` 作为顶级宿主进行承载；如果依然保留了早期的视图结构，子页面使用宿主内的路由或组件进行分发，但 UI 渲染全权交由 Compose Multiplatform 引擎。

## 3. 架构与状态管理 (MVVM)

- **多端 ViewModel**：目前项目全面拥抱官方的 Multiplatform ViewModel，因此 `ViewModel` 类可以直接在 `commonMain` 中定义并使用。
- **响应式数据流**：强约束使用 `StateFlow` 进行状态管理。UI 层不可突变（Immutable），ViewModel 内部通过 `MutableStateFlow` 维护状态，UI 层通过收集 Flow 驱动重组。

## 4. 本地数据持久化 
对于菜品、历史记录等本地数据的管理：

- **框架指定**：数据库使用支持 KMP 版本的 **Room**（Room 2.8.4），轻量可持久化数据（如一些配置和需要记忆的内容）使用 **Multiplatform Settings**（Multiplatform Settings1.2.0）。
- **代码位置**：所有的 `@Entity` 数据表结构、`@Dao` 数据访问对象以及 `@Database` 抽象类，全部放置在 `commonMain` 中运行在双端。不要向我推荐 SQLDelight，除非我明确要求。
- **数据流**：采用 database/settings -> Repository -> ViewModel -> UI

## 5. UI 美学与交互规范 (Compose)
- **动画与视觉**：对于食物随机抽取的 2 秒滚动效果，请使用 Compose 的 `Animatable` 协程动画或 `suspend` 函数配合 `Flow` 定时发射来修改文本状态，以确保高帧率。
- **布局一致性**：要求 UI 代码整洁对称。例如在水平布局 `Row` 中处理头像与食物名称时，必须熟练使用 `Alignment.CenterVertically` 参数确保跨端视觉完美居中。
- **参数说明要求**：当提供 UI 组件或新功能的实现代码时，必须列出使用的方法名，并在注释中简明扼要地标明各参数的作用。

## 6. AI 搜索要求
由于 Compose Multiplatform 技术栈迭代极快，遇到不确定的依赖库（如跨端图片加载 Coil3、跨端网络 Ktor）的 API 细节时，必须主动进行联网搜索，不准依靠旧有记忆胡编乱造。

## 7. 依赖引入要求

- 引入任何新依赖前，必须明确告知用户依赖名称、用途和影响。
- 优先复用项目现有依赖及版本目录（Version Catalog）中统一管理的版本，避免无必要地新增或手动覆盖版本。
- 不得为了简单功能引入体积较大或职责重叠的依赖。
- 对不确定或更新较快的库及 API，必须主动查询官方文档，不依靠过时记忆推测用法。
- 未经用户确认，不得擅自引入新的数据库、网络、图片加载、依赖注入或其他基础框架。

## 8. 编码与命名规范

- 包名、变量名、函数名和类名必须与仓库现有 Kotlin/KMP 风格保持一致，不混用其他语言或框架的命名习惯。
- 类名使用 `UpperCamelCase`，方法和变量使用 `lowerCamelCase`，常量使用 `UPPER_SNAKE_CASE`。
- 命名应直接表达业务含义，避免无意义缩写以及 `data`、`info`、`manager` 等过度宽泛的名称。
- 保持代码简单清晰，不编写重复逻辑、无用注释或与实现不一致的文档。
- 使用项目现有格式；除非用户明确要求，不对无关文件进行批量格式化。
- 注释用于解释业务规则、设计原因或特殊边界，不重复描述代码表面行为。

## 9. 指令与文件操作规范

禁止执行以下破坏性指令：

- `del /s`
- `rd /s`
- `rmdir /s`
- `Remove-Item -Recurse`
- `rm -rf`

同样禁止使用 Python、Java 或其他脚本绕过上述限制进行递归删除。

- 禁止对任何系统根目录、磁盘根目录或系统级目录执行删除、清理或修改操作。
- 未经用户明确授权，禁止执行跨出当前项目根目录的写入、移动、覆盖或删除操作。
- 删除、覆盖或迁移项目内已有文件前，必须先检查内容；若与任务描述不符，应先向用户说明。
- 不得修改构建产物、IDE 缓存或依赖缓存来代替修改源代码。
- 对不可逆操作、数据库数据变更及对外发布操作，执行前必须获得明确授权。

## 10. 文件规范

- 单个源文件不宜过长，原则上尽量控制在 250 行以内。
- 文件接近或超过限制时，应按职责拆分，不能仅为减少行数进行无意义拆分。
- 一个文件应聚焦单一职责，避免将 Compose UI、ViewModel、Repository 和数据模型堆放在同一文件中。
- 新增文件时应放入职责匹配的包，禁止创建含义模糊的 `utils` 或 `common` 大杂烩目录。

## 11. 新功能计划规范

- 提出修改或新增功能时，若业务规则、接口行为、数据结构或异常处理存在歧义，必须先追问。
- 不可自行虚构需求、默认值、字段、接口、角色权限或审核逻辑。
- 所有影响实现方式的细节确认完成并获得接受后，才可开始修改代码。
- 开始实现前，应简要说明预计修改或新增的文件、实现步骤、依赖变化和验证方式。
- 若实现过程中发现原方案需要扩大范围，应暂停并说明原因，不得静默扩大修改范围。

## 12. 代码修改规范

- 修改或新增代码时，只改动完成当前任务所必需的文件。
- 如需修改用户未点名的其他文件，必须提前强调文件名称、修改原因和影响。
- 非必要不进行重构、改名、依赖升级、目录迁移或配置调整。
- 必须先阅读相关代码并遵循现有实现风格，不得用整套新写法替换已有模式。
- 不得覆盖或回退用户已经存在且与当前任务无关的修改。
- 完成后应明确列出实际修改的文件、主要变化、依赖变化和测试结果。
