# 第一版后端、匿名认证与图片审核需求

## 1. 文档说明

本文档记录 WhatToEat2 第一版后端的已确认需求，主要涵盖：

- 后端技术与部署约束；
- 匿名身份认证；
- 点评数据的身份归属与权限校验；
- 用户图片上传及审核；
- 后续升级正式登录系统时的兼容要求。

未最终确定的算法、阈值、数据库类型和第三方服务等内容统一列入“待确认事项”，不在第一版需求中擅自指定。

## 2. 项目条件与设计目标

### 2.1 运行条件

- 服务器内存为 2GB；
- 初期通常只有几十名用户；
- 后端与数据库可能部署在同一台小型云服务器上；
- 优先保证实现简单、资源可控、便于个人开发者维护。

### 2.2 第一版目标

- 用户不需要注册账号或输入密码；
- App 自动为用户申请匿名身份；
- 匿名用户可以上传图片和发表点评；
- 用户只能修改或删除属于自己的点评；
- 服务端可以撤销匿名凭证或封禁用户；
- 用户上传的图片必须经过质量与内容审核后才能公开；
- 数据结构应允许未来将匿名用户升级为正式注册用户。


## 3. 总体架构

第一版采用单体后端：

```text
Compose Multiplatform 客户端
        ↓ HTTPS
Spring Boot 单体后端
        ├── 匿名认证与权限校验
        ├── 菜品、点评和图片业务
        ├── 图片审核任务管理
        └── 图片分析组件（具体实现待确认）
        ↓
关系型数据库 + 图片文件存储
```

### 3.1 后端技术

- 使用 Spring Boot；
- 使用 Spring MVC 提供 HTTP API；
- 可以使用 Spring Security 实现统一的 Token 认证过滤；
- 第一版不要求使用 WebFlux；
- Spring Boot 负责业务、认证、权限、审核状态和数据管理；
- 图片分析算法应与核心业务代码保持清晰边界，避免算法故障直接破坏业务数据。

### 3.2 资源控制原则

- JVM、数据库连接池、日志和图片分析任务均需要限制资源占用；
- 图片分析默认低并发执行，避免多张大图同时解码导致内存不足；
- 不在 JVM 内长期保留完整图片的 `ByteArray`；
- 不将图片二进制长期直接存入关系型数据库；
- 必须配置日志轮转和临时图片清理机制；
- 图片上传与匿名身份创建接口必须进行频率限制。

## 4. 用户模型

匿名用户与未来的正式用户共用统一的 `users` 表。

```text
users
- id
- type
- status
- created_at
```

字段含义：

- `id`：服务端生成的稳定用户标识；
- `type`：用户类型，第一版固定为 `anonymous`，未来可变为 `registered`；
- `status`：用户状态，例如 `active`、`banned`、`deleted`；
- `created_at`：用户创建时间。

业务数据只能关联稳定的 `users.id`，不能关联可能变化或撤销的 Token。

## 5. 匿名身份认证

### 5.1 认证方案

第一版采用：

> 固定匿名 `userId` + 单个长期不透明 Token。

Token 的规则：

- Token 由服务端通过密码学安全随机数生成；
- Token 应至少包含 32 字节随机数据，并使用适合 HTTP 传输的格式编码；
- Token 不包含用户信息，不能由 Token 内容直接推导出 `userId`；
- Token 第一版不定期自动轮换；
- Token 长期有效，直到服务端主动撤销、用户被封禁或身份被删除；
- 客户端不得自行生成或指定 Token。

### 5.2 会话数据

```text
sessions
- id
- user_id
- token_hash
- created_at
- last_used_at
- revoked_at
```

要求：

- `user_id` 关联 `users.id`；
- 服务端数据库不得保存 Token 原文；
- `token_hash` 保存 Token 的 HMAC-SHA-256 结果；
- HMAC 使用的服务端密钥放在环境变量或专用密钥配置中；
- 服务端密钥不得写入数据库、源码仓库或日志；
- `token_hash` 应建立唯一索引；
- `revoked_at` 不为空时，该 Token 不再有效。

### 5.3 匿名身份创建

客户端在本地没有 Token 时调用：

```http
POST /api/auth/anonymous
```

服务端必须在同一事务中：

1. 创建 `type = anonymous` 的用户；
2. 生成长期随机 Token；
3. 计算 Token 的 HMAC-SHA-256；
4. 保存会话记录及其关联的 `userId`；
5. 通过 HTTPS 将 Token 原文返回客户端一次。

如果事务失败，不得留下无效用户或不完整会话。

### 5.4 客户端 Token 使用

客户端将 Token 原文保存在安全存储中，并在需要身份的请求中发送：

```http
Authorization: Bearer <token>
```

要求：

- Token 不放入 URL 或查询参数；
- Token 不写入客户端日志、服务端日志、代理日志和崩溃报告；
- Android 端应使用 Keystore 支持的安全存储；
- iOS 端应使用 Keychain；
- KMP 客户端需要通过跨平台抽象访问安全存储；
- 如果需要引入新的安全存储依赖，实施前必须明确说明并确认。

### 5.5 服务端身份确认

服务端收到 Token 后：

1. 读取 `Authorization` 请求头；
2. 对 Token 计算相同的 HMAC-SHA-256；
3. 根据 `token_hash` 查询会话；
4. 检查会话未撤销；
5. 检查用户状态为 `active`；
6. 从会话记录取得可信的 `userId`。

随机伪造、数据库中不存在或已被撤销的 Token 返回 `401 Unauthorized`。

服务端不得使用请求体、路径或查询参数中的 `userId` 作为当前登录身份。

### 5.6 Token 丢失

第一版不提供匿名身份找回能力。

如果 Token 因卸载 App、清除数据或安全存储损坏而丢失：

- 服务端无法恢复原 Token；
- 用户无法继续修改原匿名身份创建的点评；
- 用户确认后可以创建新的匿名身份；
- 客户端收到 `401` 时不得无提示地反复创建新身份。

## 6. 点评权限

### 6.1 创建点评

创建点评时，客户端只提交点评业务数据，不提交用于决定归属的 `userId`。

服务端必须使用认证 Token 得到的 `userId` 写入：

```text
reviews.user_id = authenticatedUserId
```

### 6.2 修改和删除点评

用户只能修改或删除属于自己的点评。

权限判断必须同时包含：

- 目标点评 ID；
- 从有效 Token 得到的当前 `userId`。

建议在数据库更新或删除条件中直接校验所有权：

```sql
UPDATE reviews
SET ...
WHERE id = :reviewId
  AND user_id = :authenticatedUserId;
```

```sql
DELETE FROM reviews
WHERE id = :reviewId
  AND user_id = :authenticatedUserId;
```

不得相信客户端传来的 `userId`，也不得只根据 `reviewId` 执行修改或删除。

## 7. 图片上传与审核

### 7.1 审核目标

服务端需要对用户上传的图片进行以下分析：

- 图片是否模糊；
- 色彩饱和度及相关色彩质量；
- 图片是否包含不良内容；
- 后续增加的自定义图片指标。

客户端可以进行上传前提示或预检查，但客户端结果不能作为最终审核依据，服务端必须重新校验和分析。

### 7.2 基础安全校验

服务端必须校验：

- 实际文件格式，不只信任文件扩展名或 `Content-Type`；
- 文件大小；
- 图片宽高和总像素数；
- 动图或多帧图片的帧数；
- 文件能否安全解码；
- 文件名和存储路径是否合法。

上传大小、最大像素数、支持格式、每次上传数量和用户频率限制均应可配置，具体数值待确认。

### 7.3 传统图像质量指标

模糊度、饱和度以及后续自定义指标可以使用 OpenCV 等成熟图像处理库实现。

要求：

- 分析前按统一规则缩放图片，减少分辨率差异对指标的影响；
- 每项指标输出原始分数和最终判断；
- 阈值必须通过真实业务图片校准，不能直接采用未经验证的通用阈值；
- 指标与阈值应可配置，便于后续调整；
- 第一版不要求自行重写图片解码、缩放或基础视觉算法。

### 7.4 不良内容检测

不良内容检测不能只依靠肤色、饱和度等简单规则。

候选实现包括：

- 第三方图片内容审核服务；
- 部署在本地的轻量审核模型；
- 两者结合的分层审核方案。

具体采用哪种方案尚未确定。确定前需要评估：

- 审核类别和准确率；
- 用户隐私与图片数据是否发送给第三方；
- 调用费用；
- 模型常驻内存和单次推理峰值；
- CPU 推理耗时；
- 误判处理方式；
- 第三方服务或本地模型故障时的降级策略。

### 7.5 审核状态

图片上传后，在审核完成前不得直接向其他用户公开。

建议状态：

```text
PENDING
PROCESSING
APPROVED
REJECTED
REVIEW_REQUIRED
PROCESSING_FAILED
```

基本流程：

```text
上传临时图片
    ↓
基础安全校验
    ↓
质量指标分析
    ↓
不良内容检测
    ↓
通过 / 拒绝 / 待复核 / 处理失败
```

审核失败或结果不确定时，不应立即永久删除原图。具体保留期限和人工复核流程待确认。

### 7.6 图片分析组件

图片分析可以使用 C++、Java/Kotlin 或其他适合的实现，但第一版不预先强制指定语言。

如果采用 C++：

- 优先调用 OpenCV、ONNX Runtime 等成熟库；
- 不从零实现已有的图片解码和推理能力；
- 图片分析应与 Spring Boot 业务层保持接口隔离；
- 需要限制分析进程的 CPU、内存、并发数和处理时长；
- 原生组件崩溃不得导致已提交的业务数据损坏。

## 8. 安全与运维要求

- 所有携带 Token 和上传图片的接口必须使用 HTTPS；
- 对匿名身份创建、上传、点评和审核相关接口实施限流；
- 服务端不得在任何日志中记录完整 `Authorization` 请求头；
- 管理功能与普通匿名用户功能必须分权；
- 管理端应能撤销指定会话或封禁用户；
- 对数据库和图片文件进行定期备份；
- 备份不得只保存在同一台服务器上；
- 定期清理未绑定、审核失败或超期的临时图片；
- 图片分析任务必须限制并发，初始值应根据 2GB 服务器实测确定；
- 服务异常时不得通过无限自动重启掩盖 OOM 或图片处理故障。

## 9. 后续正式登录兼容性

未来新增正式登录时，增加登录身份表：

```text
auth_identities
- id
- user_id
- provider
- provider_user_id
- credential_data
- created_at
```

匿名用户绑定正式登录方式时：

1. 验证当前匿名 Token；
2. 验证新的登录凭据；
3. 将登录身份绑定到当前 `users.id`；
4. 将 `users.type` 从 `anonymous` 改为 `registered`。

用户的 `userId` 不变，因此原点评、图片和其他业务数据不需要迁移。

如果当前匿名身份要登录一个已存在的正式账号，则需要额外制定两个用户之间的数据合并与冲突处理规则，该规则不属于第一版范围。

## 10. 待确认事项

实施前还需要确认：

1. 使用 PostgreSQL、MySQL 还是其他关系型数据库；
2. 图片存储在本机文件系统还是对象存储；
3. 支持的图片格式、单张大小、最大像素数和每次上传数量；
4. 模糊度、饱和度及其他质量指标的具体算法和阈值；
5. “不良内容”的具体审核类别；
6. 不良内容检测使用第三方服务还是本地模型；
7. 图片分析组件是否采用独立 C++ 常驻进程；
8. 审核不确定时是否提供人工复核；
9. 被拒绝、处理失败及未绑定临时图片的保留期限；
10. 客户端安全存储采用自行封装的 `expect/actual` 还是新的跨平台依赖；
11. Token 丢失提示和创建新匿名身份的具体交互；
12. 服务器中 Spring Boot、数据库和图片分析组件的最终内存配额。
